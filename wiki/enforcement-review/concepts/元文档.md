# 元文档 — 生态环境执法案卷评查知识库顶层约定

> 本文件是知识库的“元文档”，定义了知识库的架构、文件命名、维护纪律及LLM角色分工。LLM在每次执行Ingest、Query、Lint等操作前，必须读取本文件及raw/SCHEMA.md，以确保操作符合统一规范。

## 一、知识库三层架构（必须坚守）

本知识库采用严格的三层架构，每一层有明确的维护主体和变更纪律：

1.  **Raw 源层（人类策展 / LLM 只读不改）**：
    -   **定位**：权威原始资料的最终真相源。所有法律全文、法典原文、案例原文、裁量基准原文、监测条例原文均存放于此。
    -   **标注**：所有Raw层文件必须包含 `status: 源真` 标签。LLM仅可引用，**严禁**在此层进行任何撰写、修改或删除操作。
    -   **路径**：`raw/` 目录下。

2.  **Wiki 层（LLM 拥有 / 全权维护）**：
    -   **定位**：本知识库的核心工作层。所有经过LLM分析、总结、关联、评查的文档均存放于此。
    -   **标注**：文件 `status` 字段为 `现行`、`草案`、`待审` 等。LLM在此层拥有全权维护权限，包括创建、修改、重组、删除文件。
    -   **路径**：`01_~09_` 目录下。

3.  **Schema 层（LLM 必须遵守）**：
    -   **定位**：定义知识库的元数据规范、文件命名规则、标签体系、触发词约定等。本文件及 raw/SCHEMA.md 均属此层。
    -   **标注**：文件 `type` 字段为 `schema` 或 `meta`。LLM必须遵守此层所有约定，但**不得**修改此层文件（除非人类明确授权）。

## 二、文件命名与目录结构

### 2.1 目录层级
-   `raw/`：Raw 源层。
-   `01_~09_`：Wiki 层，按主题或流程编号。
-   `10_元文档/`：Schema 层，存放本文件及类似约定文件。

### 2.2 文件命名规则
-   **格式**：`[序号]-[简短英文或拼音描述].md`
-   **示例**：`01-案卷评查总则.md`，`02-处罚程序.md`，`03-证据规则.md`
-   **禁止**：文件名中不得包含空格、特殊字符（除 `-` 和 `_`）。

## 三、文件 Frontmatter 规范

每个 `.md` 文件必须包含以下 YAML frontmatter 字段：

```yaml
---
title: <文件标题>
layer: <层级编号，如 01-案卷评查>
type: <文档类型：doc, schema, meta, case, law, template>
触发词: [<逗号分隔的触发词列表>]
适用场景: [<逗号分隔的适用场景描述>]
关联法条: [<关联的法律法规名称>]
调用skill: [eco-review-kb]
风险等级: <🟢低 🟡常规 🟠敏感 🔴高危>
version: <版本号，如 1.0>
status: <状态：现行, 草案, 待审, 源真>
updated: <最后更新日期，格式 YYYY-MM-DD>
tags: [<标签1>, <标签2>, ...]
---
```

## 四、LLM 维护纪律

1.  **Ingest 流程**：
    -   读取本文件及 raw/SCHEMA.md。
    -   识别新文件应归属的层级（Raw/Wiki/Schema）。
    -   自动生成符合规范的 frontmatter。
    -   检查并修正现有文件的 frontmatter 缺失或错误。

2.  **Query 流程**：
    -   优先从 Wiki 层检索信息。
    -   如需引用 Raw 层原文，必须标注 `status: 源真` 并给出原文路径。
    -   回答时需引用关联法条，如《生态环境行政处罚办法》第十条。

3.  **Lint 流程**：
    -   检查所有文件是否包含完整 frontmatter。
    -   检查 `status` 字段是否与所在层级匹配（Raw 层必须为 `源真`）。
    -   检查 `tags` 是否包含层级标签（如 `01-案卷评查`）。
    -   检查 `触发词` 是否准确反映文件内容。
