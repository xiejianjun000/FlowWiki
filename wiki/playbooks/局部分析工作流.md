# 局部分析工作流

## 概述

当问题局限于单一系统或数据域时，使用本工作流进行快速定位和修复。

## 步骤

1. **问题定位**：缩小到具体模块/组件
2. **日志分析**：提取该模块的操作日志
3. **异常识别**：对比正常基线，识别异常点
4. **原因分析**：分析异常的直接原因
5. **修复验证**：执行修复并验证效果

## 注意事项

- 确保基线数据的可靠性
- 修复后需持续监控 24 小时
- 记录修复过程到知识库

## 各步详解

### 步骤1：问题定位

缩小问题范围的方法：

- **漏斗式下钻**：从全局指标 → 集群/机房 → 服务 → 接口 → 实例逐步下钻
- **占比分析**：按错误码、接口、地区等维度统计异常占比，定位主要贡献者
- **差分诊断**：对比正常时段和异常时段的请求分布差异
- **变更锚定**：检查最近一次变更涉及的具体模块/配置文件

目标：在 10 分钟内将问题锁定到具体模块或组件。

### 步骤2：日志分析

- **错误日志优先**：先过滤 ERROR/FATAL 级别日志，提取异常堆栈
- **请求追踪**：通过 trace_id/request_id 追踪单次请求的完整生命周期
- **关键字搜索**：使用模块名、类名、方法名作为搜索关键词
- **日志模式识别**：用聚类算法对日志进行模式分组，发现共性错误
- **慢查询分析**：检查数据库慢查询日志和接口响应时间分布

### 步骤3：异常识别

与基线对比的具体维度：

| 对比维度 | 基线来源 | 异常判定 |
|---------|---------|---------|
| 指标值 | 同比上周同时段 | 偏离 > 3 倍标准差 |
| 错误率 | 过去 7 天均值 | 超过 2% 且持续 5 分钟 |
| 延迟 P99 | 过去 1 小时均值 | 超过基线的 200% |
| 日志量 | 日均日志量 | 突增 > 50% 或骤降 > 30% |
| 调用量分布 | 过去 24 小时分布 | 卡方检验 p < 0.01 |

### 步骤4：原因分析

直接原因定位技术：

1. **对比法**：A/B 对比正常和异常的请求参数、返回结果差异
2. **二分排查**：对可疑代码段使用二分注释法/灰度发布逐步缩小范围
3. **依赖检查**：逐个检查下游依赖（数据库、缓存、消息队列）的健康状态
4. **配置比对**：diff 当前配置与上一版本配置的差异
5. **线程栈分析**：获取异常时刻的 JVM/Goroutine 线程 dump 分析

### 步骤5：修复验证

- **灰度验证**：先在 1-5% 流量上验证修复效果，确认无副作用后全量生效
- **A/B 对照**：保留部分流量不修复作为对照组，观察指标差异
- **回归检查**：验证修复后相关接口和下游服务的指标是否正常
- **监控静默确认**：修复后告警是否自动恢复，并保持 30 分钟无复发

## 快速诊断工具箱

| 问题类型 | 诊断命令/工具 | 关键指标 |
|---------|-------------|---------|
| CPU 突增 | top / htop / perf | CPU 使用率、top 进程 |
| 内存泄漏 | jmap / heap dump / pprof | 堆内存增长趋势、GC 频率 |
| 数据库慢查询 | pt-query-digest / 慢查询日志 | 慢查询数、锁等待时间 |
| 网络超时 | tcpdump / ss / netstat | TIME_WAIT 连接数、丢包率 |
| 磁盘 IO | iostat / iotop | IO util%、await 延迟 |
| 应用错误 | 异常堆栈 / APM (如 SkyWalking) | 错误数、错误率趋势 |

## 输出物

局部分析完成后应产出的文档：

1. **分析报告**：问题描述 → 定位过程 → 直接原因 → 修复措施 → 验证结果
2. **时间线**：关键事件和操作的精确时间线
3. **监控增强建议**：此类问题是否需要新增监控/告警规则
4. **知识沉淀**：将典型问题和处理方法记录到故障知识库
