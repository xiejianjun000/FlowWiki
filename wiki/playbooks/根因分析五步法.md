# 根因分析五步法

## 概述

根因分析五步法是一种系统化的问题诊断流程，从异常现象出发，逐步深入到根本原因，并给出修复建议。

## 步骤

### 第一步：现象描述

收集异常现象的完整描述，包括发生时间、影响范围、表现特征。

### 第二步：数据收集

调取相关日志、审计记录、系统指标，构建完整的事件时间线。

### 第三步：假设生成

基于时间线和领域知识，生成可能的原因假设列表。

### 第四步：假设验证

逐个验证假设，通过数据对比、复现实验或排除法缩小范围。

### 第五步：根因确认与建议

确认根本原因，给出修复建议和预防措施。

## 注意事项

- 避免过早收敛到单一假设
- 注意多重因果的可能性
- 记录排除的假设及排除理由

## 每步详解

### 第一步详解：现象描述

现象描述的质量直接决定后续分析的方向。应收集以下要素：

| 要素 | 示例 | 获取途径 |
|------|------|---------|
| 异常指标 | 订单成功率从 99.5% 降至 92.3% | 监控告警、用户反馈 |
| 起止时间 | 2026-07-17 14:23 ~ 持续中 | 告警时间戳、日志时间范围 |
| 影响范围 | 华东区用户、支付模块 | 用户报障、拨测结果 |
| 触发事件 | 代码发布 v2.3.1、配置变更 | 变更管理平台、审计日志 |
| 异常形态 | 瞬时下跌 vs 持续恶化 vs 间歇性抖动 | 监控曲线图分析 |

### 第二步详解：数据收集

- **系统日志**：应用日志、中间件日志、操作系统日志，按时间窗口过滤
- **监控指标**：CPU、内存、网络 IO、QPS、延迟、错误率等时序数据
- **变更记录**：代码部署、配置变更、数据库变更、流量调度记录
- **审计追踪**：操作审计日志，核对敏感操作的时间是否与异常时间吻合
- **关联事件**：上游系统告警、第三方服务状态、网络故障公告

### 第三步详解：假设生成

常用假设生成技术：

1. **鱼骨图 (Ishikawa)**：从人、机、料、法、环五个维度枚举可能原因
2. **5-Why 分析**：对现象连续追问 5 次"为什么"，逐层深入
3. **变更倒查**：列出异常时间窗口内的所有变更，逐项评估嫌疑等级
4. **历史类比**：检索历史故障数据库，匹配相似模式获取候选原因

### 第四步详解：假设验证

| 验证方法 | 适用场景 | 成本 |
|---------|---------|------|
| 数据对比 | 假设可用历史数据证伪 | 低 |
| 回滚对照 | A/B 测试或灰度回滚 | 中 |
| 沙箱复现 | 在隔离环境中重现条件 | 中 |
| 代码审查 | 代码 Bug 相关假设 | 低 |
| 压力测试报告 | 容量/性能相关假设 | 高 |

### 第五步详解：根因确认与建议

输出物应包含：

1. **根因声明**：一句话描述根本原因
2. **因果链路图**：从根因到现象的完整因果链
3. **修复方案**：立即止血措施 + 长期根治方案
4. **预防措施**：监控增强 + 流程改进 + 自动化校验
5. **故障复盘 (Postmortem)**：时间线、影响、根因、改进项

## 常用工具

| 工具 | 用途 | 特点 |
|------|------|------|
| Grafana | 监控数据可视化与对比 | 支持多数据源叠加分析 |
| Kibana | 日志聚合搜索 | 全文检索、时间线视图 |
| Jupyter | 交互式数据探索 | Python/R 灵活分析 |
| Jaeger/Zipkin | 分布式链路追踪 | 调用链可视化 |
| 故障管理系统 (如 PagerDuty) | 故障记录与复盘归档 | 知识沉淀和搜索 |

## 常见陷阱

1. **确认偏误**：只找支持自己假设的证据，忽略反例 → 对策：刻意寻找反证
2. **锚定效应**：过度依赖初步线索而忽略后续发现 → 对策：保持假设列表开放状态
3. **归因错误**：将相关性误判为因果性 → 对策：检查是否有混淆变量
4. **过早止步**：找到表面原因就停止 → 对策：坚持完成 5-Why 直到找到系统性根因
5. **责备文化**：将根因归结于个人而非系统缺陷 → 对策：采用"无责备"的事后分析文化
