# 跨域分析工作流

## 概述

当问题涉及多个业务系统或数据域时，需要跨域协同分析。本工作流定义了跨域根因分析的标准流程。

## 步骤

1. **域识别**：确定涉及的系统/数据域
2. **关联映射**：建立跨域数据关联关系
3. **时间对齐**：对齐不同域的时间戳
4. **因果推断**：跨域追踪因果链路
5. **综合报告**：输出跨域分析结论

## 注意事项

- 不同域的时间精度可能不一致
- 注意数据权限和访问控制
- 跨域因果推断需要领域专家确认

## 各步详解

### 步骤1：域识别

跨域分析的起点是准确界定问题波及的系统范围。域识别通常通过以下方式完成：

- **告警关联分析**：同一时间段内多个系统的告警可能存在因果关联
- **调用链拓扑图**：从分布式追踪系统导出上下游依赖关系，标记异常节点
- **数据流图审查**：检查 ETL 管道和消息队列的吞吐量变化，定位数据传播路径
- **业务映射表**：维护业务功能到系统组件的映射，如"订单支付→支付网关→账务系统→风控引擎"

### 步骤2：关联映射

建立跨域数据关联时需明确：

| 关联类型 | 示例 | 连接键 |
|---------|------|--------|
| 用户维度 | 同一用户的订单和支付记录 | user_id |
| 时间维度 | 同一时段的订单和库存扣减 | timestamp |
| 流水维度 | 支付流水和账务流水 | transaction_id |
| 事件维度 | 登录事件和风控事件 | session_id |

注意：不同系统对同一实体的 ID 编码可能不同，可能需要通过映射表或哈希函数进行 ID 转换。

### 步骤3：时间对齐

- **时钟偏差检测**：不同服务器的 NTP 同步偏差可能导致几分钟的时间差，应先校准
- **时区统一**：所有时间戳统一转为 UTC 或业务本地时间
- **时间粒度对齐**：秒级日志 vs 分钟级监控 → 按粗粒度假定对齐或细粒度插值
- **时间窗选择**：建议取异常前 30 分钟到异常结束后 15 分钟的滑动窗口

### 步骤4：因果推断

跨域因果推断的核心技术：

1. **Granger 因果检验**：如果域 A 的历史数据能提高对域 B 的预测精度，则 A 是 B 的 Granger 原因（注意：这仍是统计推断，非真实因果）
2. **事件序列分析**：按时间顺序排列多域的关键事件，寻找因果传导路径
3. **反事实推理**：假设移除某个域的事件，其他域是否仍会异常
4. **专家验证**：跨域结论必须由各域负责人确认，防止"证真偏差"

### 步骤5：综合报告

跨域报告的必备要素：
- 跨域事件时间线图（标注每个域的关键事件）
- 因果传导链路图（箭头表示因果关系方向）
- 各域的影响评估和恢复时间
- 跨域协作改进建议（如监控联动、告警收敛）

## 跨域分析工具链

| 工具 | 用途 |
|------|------|
| Jaeger / Zipkin | 分布式调用链可视化 |
| ELK Stack | 多源日志联合查询和时间线分析 |
| Grafana Dashboard | 多域指标叠加对比 |
| Apache Kafka / Pulsar | 消息队列的 backlog 和消费延迟分析 |
| Confluence / 语雀 | 跨团队文档协作和评审
