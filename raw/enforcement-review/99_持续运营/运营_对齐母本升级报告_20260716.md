# 执法督察评查知识库 · 对齐母本设计方案升级报告（2026-07-16）

## 一、起因
用户指向「企业合规AI管理知识库」中的《Obsidian×LLM Wiki 知识库设计方案》（基于企业环保合规AI管家白皮书·七层架构+12Skills+五大场景）。该方案是我建 `执法督察评查知识库` 所依方法论的**母本规范**。

## 二、差距诊断
我建的库在「原子化/双向链接/Skill路由/提示词」已对齐，但缺母本最关键的工程纪律：
1. M0 元文档层缺失（设计原则/标签/命名/法典切换/审计版本）
2. frontmatter 缺 `layer/type/version/status/tags` 统一 schema
3. 无「AI读取顺序协议」
4. 无每目录 MOC / 统一命名规范

## 三、升级动作（低风险，原有27篇内容不动）
1. **升级 `_templates/单篇文档模板.md`**：frontmatter 增加 layer/type/version/status/tags 5字段 + 字段说明注释。
2. **新增 `10_元文档/` M0层（5篇）**：
   - 元_设计原则（LLM Wiki五条 + 本库四条红线）
   - 元_标签体系（层/风险/领域/类型/状态 五类标签 + LLM读取顺序）
   - 元_命名规范（目录/文件/frontmatter 命名一致性）
   - 元_法典切换预案（🔴 8-15临界切换纪律，一级预警）
   - 元_审计与版本管理（版本/审计/证据链/Git）
3. **批量注入 frontmatter 字段**：用脚本为 27 篇业务文档补齐 layer/type/version/status/tags（按目录推断默认值），正文零改动。
4. **MOC 升级**：00-能力地图 新增「八、AI读取顺序协议」+「九、元文档层」两节，对齐母本第九节遍历顺序。

## 四、对齐结果
- 文档总数：33 篇（32 业务 + 1 模板）
- 全部业务文档含统一 frontmatter schema（layer/type/version/status/tags）
- 目录结构对齐母本「00-首页/10-元文档/.../90-场景/99-运营」思路（本库聚焦评查单域，用 00-06 + 10_元文档 对应）
- 确立「AI读取顺序协议」：MOC→文档→SOP→法条指针→督察→提示词→预警/元文档

## 五、仍可后续补强（非阻塞）
- 25项清单第15-25项目前为骨架（旧库 07_ 有原文）
- 按日/查封/行政拘留办法的旧库文件为占位，法条号可精确化
- 可选：安装母本推荐的 Dataview/Templater/Git 插件，启用 Bases 数据库视图
- 可选：把本库接入 Git 私有仓库做版本管理（元_审计与版本管理 已规定提交格式）
